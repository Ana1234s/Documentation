3 # PocketMine-MP Contribution Guidelines
4 
5 You must follow these guidelines if you wish to contribute to the PocketMine-MP code base, or participate in issue tracking.
6 
7 ## I have a question
8 * For questions, please refer to the _#pocketmine_ or _#mcpedevs_ IRC channel on Freenode. There is a [WebIRC](http://webchat.freenode.net?channels=pocketmine,mcpedevs&uio=d4) if you want.
9 * You can ask directly to _[@PocketMine](https://twitter.com/PocketMine)_ in Twitter, but don't expect an immediate reply.
10 * You may use our [Forum](http://forums.pocketmine.net) to ask questions.
11 * We do not accept questions or support requests in our issue tracker.
12 
13 ## Creating an Issue
14  - First, use the [Issue Search](https://github.com/PocketMine/PocketMine-MP/search?ref=cmdform&type=Issues) to check if anyone has reported it.
15  - If your issue is related to a plugin, you must contact their original author instead of reporting it here.
16  - If your issue is related to a PocketMine official plugin, or our Android application, you must create an issue on that specific repository.
17  - **Support requests are not bugs.** Issues such as "How do I do this" are not bugs and are closed as soon as a collaborator spots it. They are referred to our Forum to seek assistance.
18  - **No generic titles** such as "Question", "Help", "Crash Report" etc. If an issue has a generic title they will either be closed on the spot, or a collaborator will edit it to describe the actual symptom.
19  - Information must be provided in the issue body, not in the title. No tags are allowed in the title, and do not change the title if the issue has been solved.
20  - Similarly, no generic issue reports. It is the issue submitter's responsibility to provide us an issue that is **trackable, debuggable, reproducible, reported professionally and is an actual bug**. If you do not provide us with a summary or instructions on how to reproduce the issue, it is a support request until the actual bug has been found and therefore the issue is closed.

22 ## Contributing Code
23 * Use the [Pull Request](https://github.com/PocketMine/PocketMine-MP/pull/new) system, your request will be checked and discussed.
24 * __Create a single branch for that pull request__
25 * Code using the syntax as in PocketMine-MP. See below for an example.
26 * The code must be clear and written in English, comments included.
27 * Use descriptive commit titles
28 * __No merge commits are allowed, or multiple features per pull request__
29 
30 **Thanks for contributing to PocketMine-MP!**

32 ### Code Syntax
33 
34 It is mainly [PSR-2](https://github.com/php-fig/fig-standards/blob/master/accepted/PSR-1-basic-coding-standard.md) with a few exceptions.
35 * Opening braces MUST go on the same line, and MUST NOT have spaces before.
36 * `else if` MUST be written as `elseif`. _(It is in PSR-2, but using a SHOULD)_
37 * Control structure keywords or opening braces MUST NOT have one space after them.
38 * Code MUST use tabs for indenting.
39 * Long arrays MAY be split across multiple lines, where each subsequent line is indented once.
40 * Files MUST use only the `<?php` tag.
41 * Files MUST NOT have an ending `?>` tag.
42 * Code MUST use namespaces.
43 * Strings SHOULD use the double quote `"` except when the single quote is required.
44 * Argument lists MAY NOT be split across multiple lines, except long arrays.

76 ### RFC and Voting
77 * These are big Pull Requests or contributions that change important behavior.
78 * RFCs will be tagged with the *PR: RFC* label
79 * A vote will be held once the RFC is ready. All users can vote commenting on the Pull Request
80 * Comments MUST use "Yes" or "No" on the FIRST sentence to signify the vote, except when they don't want it to be counted.
81 * If your comment is a voting comment, specify the reason of your vote or it won't be counted.
82 * After voting has been closed, no further votes will be counted.
83 * An RFC will be rejected if less than 50% + 1 (simple majority) has voted Yes.
84 * If the RFC is approved, Team Members have the final word on its implementation or rejection.
85 * RFCs with complex voting options will specify the vote percentage or other details.

90 ### Labels
91 To provide a concise bug tracking environment, prevent the issue tracker from over flowing and to keep support requests out of the bug tracker, PocketMine-MP uses a label scheme a bit different from the default GitHub Issues labels.
92 
93 PocketMine-MP uses GitHub Issues Labels. There are a total of 12 labels.
94 
95 Note: For future reference, labels must not be longer than 15 letters.
96 
97 #### Categories
98 Category labels are prefixed by `C:`. Multiple category labels may be applied to a single issue(but try to keep this to a minimum and do not overuse category labels).
99  - C: Core - This label is applied when the bug results in a fatal crash, or is related to neither Gameplay nor Plugin API.
100  - C: Gameplay - This label is applied when the bug effects the gameplay.
101  - C: API - This label is applied when the bug effects the Plugin API.
102 
103 #### Pull Requests
104 Pull Requests are prefixed by `PR:`. Only one label may be applied for a Pull Request.
105  - PR: Bug Fix - This label is applied when the Pull Request fixes a bug.
106  - PR: Contribution - This label is applied when the Pull Request contributes code to PocketMine-MP such as a new feature or an improvement.
107  - PR: RFC - Request for Comments
108 
109 #### Status
110 Status labels show the status of the issue. Multiple status labels may be applied.
111  - Reproduced - This label is applied when the bug has been reproduced, or multiple people are reporting the same issue and symptoms in which case it is automatically assumed that the bug has been reproduced in different environments.
112  - Debugged - This label is applied when the cause of the bug has been found.
113  - Priority - This label is applied when the bug is easy to fix, or if the scale of the bug is global.
114  - Won't Fix - This label is applied if the bug has been decided not be fixed for some reason. e.g. when the bug benefits gameplay. *This label may only be applied to a closed issue.*
115 
116 #### Miscellaneous
117 Miscellaneous labels are labels that show status not related to debugging that bug. The To-Do label and the Mojang label may not be applied to a single issue at the same time.
118  - To-Do - This label is applied when the issue is not a bug, but a feature request or a list of features to be implemented that count towards a milestone.
119  - Mojang - This label is applied when the issue is suspected of being caused by the Minecraft: Pocket Edition client, but has not been confirmed.
120  - Invalid - This label is applied when the issue is reporting a false bug that works as intended, a support request, etc. *This label may only be applied to a closed issue.*
121 
122 ### Closing Issues
123 To keep the bug tracker clear of non-related issues and to prevent it from overflowing, **issues must be closed as soon as possible** (This may sound unethical, but it is MUCH better than having the BUG TRACKER filled with SUPPORT REQUESTS and "I NEED HELP").
124 
125 If an issue does not conform to the "Creating an Issue" guidelines above, the issue should be closed.

127 ### Milestones
128 PocketMine-MP uses GitHub Milestones to set a goal for a new release. A milestone is set on the following occasions.
129 
130  - A new Beta release
131  - A new Stable release
132 
133 A milestone must use the following format:
134 ```
135 Alpha_<version_number> [release_title][release_version]
136 ```
137 For example:
138 ```
139 Alpha_1.4 beta2
140 ```
